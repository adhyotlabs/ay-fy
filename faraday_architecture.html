NmapRunner Scan — End to End Data Flow

Trace exactly what happens from clicking "Run" in the UI to seeing assets appear in the workspace.

1

User triggers scan in Browser UI

Selects NmapRunner executor, sets TARGET=["192.168.1.1"], clicks Run

Vue.js emits a WebSocket event to the /dispatcher Socket.IO namespace.

Payload: {"action":"RUN","executor":"nmap","args":{"TARGET":["192.168.1.1"]},"workspaces":["test"],"execution_ids":[13]}

The server forwards this to the connected agent dispatcher process via the same namespace.
↓
2

faraday-dispatcher receives run command

dispatcher_io.py → on_run() → check_cmds() → create_process()

check_cmds(): runs nmap --version to verify nmap is installed.

create_process(): builds command from executor config:
repo_executor: nmap.py → resolves to /path/to/.venv/bin/python3 /path/to/nmap.py

Sets env vars: EXECUTOR_CONFIG_TARGET='["192.168.1.1"]', PATH=...

Spawns subprocess via asyncio.create_subprocess_shell()
↓
3

nmap.py executor runs nmap + parses output

Calls nmap binary, parses XML with NmapPlugin, outputs JSON to stdout

nmap.py in faraday_agent_dispatcher/static/executors/official/:

1. Reads EXECUTOR_CONFIG_TARGET env var
2. Builds nmap command: nmap -oX - -- 192.168.1.1
3. Runs via subprocess.run()
4. Parses XML output with NmapPlugin from faraday-plugins
5. Prints JSON to stdout: {"hosts":[{"ip":...,"services":[...]}],"command":{...}}
↓
4

Dispatcher sends JSON to Faraday bulk_create API

StdOutLineProcessor reads stdout line → POST /_api/v3/ws/test/bulk_create

executor_helper.py StdOutLineProcessor.processing():

Reads each JSON line from nmap.py stdout, adds execution_id, POSTs to:
POST http://localhost:5985/_api/v3/ws/test/bulk_create
Header: Authorization: Agent <agent_token>

Then end_f() sends a closing empty-hosts payload to mark command complete.
↓
5

bulk_create API saves JSON to disk, queues for processing

bulk_create.py → writes temp file → REPORTS_QUEUE.put()

bulk_create.py BulkCreateView.post():

1. Validates JSON schema with Marshmallow (BulkCreateSchema)
2. Writes to ~/.faraday/uploaded_reports/<random>.json
3. Puts (workspace_name, command_id, file_path, None, user_id, False, False, None, None, None, None, None) into REPORTS_QUEUE
4. Returns HTTP 201 immediately
↓
6

Background greenlet processes the report

reports_manager_background_task() → process_report() → bulk_create() → _create_host()

reports_processor.py reports_manager_background_task():

Runs as a gevent greenlet inside the Flask server process (started at server boot).

Picks up items from REPORTS_QUEUE every 0.5s → calls process_report()
→ reads JSON file → calls send_report_data()
→ calls bulk_create(ws, command, data, data_already_deserialized=True)
→ calls _create_host() for each host → commits to PostgreSQL
→ deletes temp file
↓
7

Assets appear in the workspace UI

Host + Services committed to DB → UI polls/refreshes → Assets tab shows results

PostgreSQL now has:
• hosts table: new row for 192.168.1.1
• services table: rows for each open port (http:80, postgres:5432…)
• commands table: audit trail of the scan run

The Vue.js UI refreshes the workspace view and shows the new asset with services.

QA Testing Surface

Faraday as an Application Under Test. Key API endpoints, test types, and attack surface for each module.

🔐 Authentication APIs

  • POST/_api/login — session login (email+password)
  • POST/_api/logout — session end
  • GET/_api/v3/agent_token — get TOTP for agent registration
  • GET/_api/session — current user info
  • Test: brute force lockout, TOTP replay, session fixation, token expiry

🏢 Workspace APIs

  • GET/_api/v3/ws/ — list all workspaces
  • POST/_api/v3/ws/ — create workspace
  • PUT/_api/v3/ws/{name} — update workspace
  • DEL/_api/v3/ws/{name} — delete workspace
  • Test: tenant isolation, cross-workspace data leakage, auth bypass

🖥️ Host / Asset APIs

  • GET/_api/v3/ws/{ws}/hosts — list hosts (paginated)
  • POST/_api/v3/ws/{ws}/hosts — create host
  • GET/_api/v3/ws/{ws}/hosts/{id} — host detail
  • DEL/_api/v3/ws/{ws}/hosts/{id} — delete host
  • Test: IDOR (host_id from another workspace), mass assignment, filter injection

🚨 Vulnerability APIs

  • GET/_api/v3/ws/{ws}/vulns — list vulns
  • POST/_api/v3/ws/{ws}/vulns — create vuln
  • PUT/_api/v3/ws/{ws}/vulns/{id} — update (status, severity)
  • GET/_api/v3/ws/{ws}/vulns?severity=critical — filter
  • Test: XSS in description/name, SQL injection in filters, bulk delete safety

📦 Bulk Create (key endpoint)

  • POST/_api/v3/ws/{ws}/bulk_create
  • Body: {"hosts":[...],"command":{...},"execution_id":N}
  • Auth: Authorization: Agent <token>
  • Test: large payloads (DoS), malformed JSON, deeply nested objects, duplicate hosts, boundary values on ports

🤖 Agent APIs

  • GET/_api/v3/agents — list agents
  • POST/_api/v3/agents — register agent (needs TOTP)
  • GET/_api/v3/agents/{id}/runs — run history
  • WebSocket: ws://localhost:5985/socket.io/?ns=/dispatcher
  • Test: expired TOTP reuse, agent impersonation, WebSocket message injection

⚡ Performance / Load Tests

  • Bulk create with 1000 hosts / 10000 services in one payload
  • Concurrent workspace access from 50 simultaneous clients
  • Large vulnerability list pagination (limit/offset)
  • Report upload of 100MB nessus XML file
  • WebSocket connection pool exhaustion

🎭 UI / E2E Tests

  • Login → create workspace → run NmapRunner → verify assets appear
  • Filter vulnerabilities by severity, status, assignee
  • Export workspace to CSV/XML — verify completeness
  • Multi-user: admin creates workspace, analyst views it
  • Tools: Playwright (via MCP), Selenium, Cypress
